## 1. 技術架構（不要動原本架構）

- **純手寫 HTML + CSS + 原生 JavaScript，無框架、無建置工具（no build step）。**
- 部署在 **GitHub Pages**（repo `janny00143/chouchou-homepage`，`main` 分支，Public，push 後自動上線）。
- 線上網址：https://janny00143.github.io/chouchou-homepage/
- 不要引入 React／Vue／打包工具／npm 依賴；維持「改完直接能跑」的靜態站本質。

| 檔案 | 用途 |

## 2. 文章系統怎麼運作（最重要）

- 所有文章的**內容資料**都寫在 `index.html` 的 **`ART` 陣列**；首頁靠 JS 渲染卡片。**另外**每篇會由 `generate-pages.cjs` 產生一個獨立 SEO 靜態頁（見第 9 節）——但**文章內容仍只在 `ART` 維護**，不要手改獨立頁。
- **唯一例外：民宿 a4**，用 `url:"minpaku.html"` 讓卡片直接跳轉到獨立頁，不走 JS 文章渲染。
- 每篇文章物件欄位：`{id, cat, date, title, tags, cover, video, ex, body[]}`；`url` 為選填，有 `url` 就跳外部頁。
- `cat` 對應 `CATS` 的 6 個分類：`foreign`（外國人買房）/ `live`（自住攻略）/ `invest`（投資收租）/ `minpaku`（民宿法規）/ `area`（區域介紹）/ `loan`（貸款稅務）。

### `body` 寫法（渲染規則）
渲染邏輯：`a.body.map(p => p.trim().startsWith('<div') ? p : '<p>'+p+'</p>')`
- **純文字段落**：直接寫字串即可，JS 會自動包成 `<p>`。
- **要做彩色框／圖表**：讓字串**以 `<div` 開頭**，這樣會原樣輸出。inline style 用單引號 `'`，外層字串用雙引號 `"`（參考 a2 的寫法）。

### 其他內容陣列
`S`（站台設定／品牌／LINE／社群／公司資訊）、`CATS`（分類）、`PROCESS`、`LIVE`、`INV`、`FAQ`、`NAV`、`REVIEWS`（首頁客戶回饋）。

### 客戶回饋／心得（REVIEWS）與投稿
- 首頁「客戶怎麼說」區塊由 `index.html` 的 **`REVIEWS` 陣列**（`{who, text}`）渲染；`ja.html` 由 `build-ja-home.cjs` 的 `REVIEWS_JA` 渲染；`index-cn.html` 由 build-cn 自動轉。
- 客戶在 `feedback.html`（意見回饋頁，Web3Forms 送到周周信箱）投稿，可勾選「同意公開」並選擇**具名或匿名**；周周收到信、**確認同意後**才由 AI／周周把該則加進 `REVIEWS`（具名用客戶稱呼、匿名用「匿名客戶」），重跑產生器三語同步後刊出。**未經同意不得公開任何客戶回饋或姓名。**

## 3. 文章風格（寫／改文章請遵守）

- **⭐ 最高原則：凡事站在周周（日本仲介、網站持有者）的商業利益寫。** 周周是仲介、也是站主，內容目的是建立專業信任、導流到加 LINE 諮詢成交。所以：
  - **要**：凸顯周周的專業與價值、讓客戶覺得「找她辦最安心」、強調有中文服務／幫忙把關、最後導向加 LINE。
  - **不要**：教客戶怎麼砍仲介費、怎麼避開仲介（自售／建商直售省仲介費）、凸顯「仲介費是最大一筆／很貴」、跟別國比顯得日本收更貴——這些站在消費者省錢角度，反而傷害周周的生意。中性的制度說明可以，但不要幫客戶精算怎麼少付她錢。
- **圖表適量即可，不是每篇都要塞。** 表格／長條圖／圓餅圖／折線圖依主題「適當」使用；資料型主題可用一兩個，一般散文型文章不用硬加。
- **以部落格散文為主，少用框框**——口語、像周周本人在跟客戶聊天，先講人話再講重點。
- **特例：a2** 可保留圖文框（綠框／黃框／藍框／玫瑰漸層 💡提醒框／雙欄比較 grid）＋月付試算圖，當作「長文＋數據版」範本；其他文章不要硬塞滿框。
- **重點數字一律用 `<b>` 標粗**（例如金額、利率、成數、坪單價）。
- 視覺語彙（若要用框）沿用 a2：綠框=結論／適合、黃框=注意、藍框=補充、玫瑰漸層框=💡周周的提醒。

### 圖片尺寸規則（周周指示，務必遵守）
- **封面圖／內文插圖都不要放太大。**
- **照片類**（一般實景照）→ 封面用 260px 固定橫幅（`.acov`，比照仲介費文章）。
- **合成圖／資訊圖／含文字的圖**（如「日本門號」SIM 卡圖、防震三工法對照圖）→ **整張顯示不可裁切**（`coverFit:"contain"`），但要限制最大高度、置中，不可撐滿寬度變超大。
- **內文插圖**（body 內的 `<img>`）用 `class='artimg'`（響應式）：手機填滿寬度、電腦以 `max-height`（現 170px）限制，整張顯示、置中；`.artimg` 定義在 `index.html` 的 `<style>`（會被產生器複製到各文章頁）。
- **封面三種模式**：一般照片→省略 coverFit，用 `.acov` 210px 橫幅（背景裁切）；資訊圖（如防震對照圖 a26）→ `coverFit:"contain"`，整張顯示、max-height 210px 置中不撐滿；橫幅式合成圖（如門號 a34）→ `coverFit:"full"`，滿版 `width:100%` 整張顯示、填滿欄寬跟照片封面一致。
- 尺寸樣式集中在 `generate-pages.cjs`／`build-ja.cjs` 的 contain 封面模板，以及 `index.html` ART／`ja-content.json` 的內文 `<img>` style；改完重跑產生器並三語同步。

- **預設繁體中文（台灣用語）**——用台灣習慣的詞（例如「房貸」「頭期款」「坪」「轉售」），不要中國大陸用語。
- **網頁內容能中文化就盡量中文化**，目標是「讓不懂日文的台灣人也看得懂」。
- **常用語若是日文，採「中文在前、括號標日文原文」的格式**，例如：表面投報率（表面利回り）、實質投報率（実質利回り）、修繕基金（修繕積立金）、中古公寓大樓（マンション）、Flat 35 房貸（フラット35）。
- **沒有對應中文的專有名詞／來源／品牌名**，用「英文（日文原文）」標示，例如：Plaza Homes（プラザホームズ）、Recruit 房貸資訊（リクルート住宅ローン）、三井不動產 Rehouse（三井のリハウス）。

## 9. SEO 與文章獨立頁（改文章後必看）

為了被搜尋引擎收錄、且每篇有獨立可分享網址，除了 `index.html` 的 `ART` 資料外，**每篇文章還有一個獨立的靜態 SEO 頁**（檔名是英文 slug，例如 `japan-property-tax-guide.html`）。

- 這些頁由 **`generate-pages.cjs`** 自動產生，含專屬 `title`／`description`／`canonical`／Open Graph／JSON-LD 結構化資料；內容直接寫在 HTML（搜尋引擎抓得到）。
- 首頁卡片是真連結（`<a href>`），指向這些頁；連結用 `index.html` 裡的 **`SLUG` 對照表**（`id → slug`）。
- **改／加文章的 SOP**：
  1. 改 `index.html` 的 `ART`。
  2. （**新增**文章時）在 `generate-pages.cjs` 與 `index.html` 的 `SLUG` **兩處都加同一組 `id→slug`**（slug 用英文關鍵字）。
  3. 在 repo 根目錄執行 `node generate-pages.cjs` 重新產生獨立頁與 `sitemap.xml`／`robots.txt`。
  4. commit／push。
- **不要手改** `<slug>.html`、`sitemap.xml`、`robots.txt`（重跑產生器會覆蓋）。
- 讓 Google 收錄：須在 **Google Search Console** 驗證網站並提交 `sitemap.xml`（帳號層級操作，由周周做；驗證碼可交給 AI 加到首頁 head）。

> 註：這等於多了「產生頁面」一步。雖非傳統 build，但**改完文章一定要重跑產生器**，否則獨立頁與 sitemap 會跟 `ART` 不同步。
